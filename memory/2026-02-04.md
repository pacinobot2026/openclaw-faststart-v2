## Step 7: Memory Install - FULL IMPLEMENTATION

**Status:** ✓ COMPLETE - Updated BOOTSTRAP.md with full implementation

**User View (exact text from BOOTSTRAP.md):**
1. "Now that I've started building up my memory, I'd like to install a memory script to help me manage your projects and data more efficiently. This means fewer headaches for you."
2. "Would you like me to install my memory system?"
3. User confirms
4. "Stand by while I set this up..."
5. "All Finished! I'm now able to handle your data, projects, and skills much more efficiently."

**Backend (what actually runs):**

1. **Config patch** via gateway tool:
   - `compaction.memoryFlush.enabled: true`
   - `memorySearch.enabled: true`
   - `memorySearch.sources: ["memory", "sessions"]`
   - `memorySearch.experimental.sessionMemory: true`
   - Apply config (triggers restart)

2. **Create memory/ folder** if doesn't exist

3. **Create MEMORY.md** with starter template for long-term memory

4. **Update AGENTS.md** with journaling protocol:
   - Real-Time Journaling Protocol (BEFORE/DURING/AFTER)
   - Write steps before browser/heavy tasks
   - Update status as you go
   - Mark COMPLETE or FAILED
   - The rules WE follow so new agent follows them too

**Goal:** New agent wakes up with memory system fully configured AND knows HOW to journal properly from day one. No learning the hard way.

### ACTUAL IMPLEMENTATION (for fresh agent):

**Backend Step 1: Config Patch**
```
gateway config.patch with:
{
  "agents": {
    "defaults": {
      "memorySearch": {
        "enabled": true,
        "sources": ["memory", "sessions"],
        "experimental": {
          "sessionMemory": true
        }
      },
      "compaction": {
        "memoryFlush": {
          "enabled": true
        }
      }
    }
  },
  "hooks": {
    "internal": {
      "entries": {
        "session-memory": {
          "enabled": true
        }
      }
    }
  }
}
```

**Backend Step 2: Create memory folder**
```
exec: mkdir memory (if not exists)
```

**Backend Step 3: Create MEMORY.md template**
Write to MEMORY.md:
```markdown
# MEMORY.md - Long-Term Memory

*Last updated: [TODAY'S DATE]*

## Who I Am
- **Name:** [From IDENTITY.md]
- **Role:** AI assistant

## Who You Are
- **Name:** [From USER.md]
- **Notes:** [Summary from profile]

## Important Lessons Learned
*(Add lessons as you learn them)*

## Key Workflows
*(Document workflows as you master them)*

## Current Projects
*(Track active work)*

---

*This file is my curated long-term memory. Updated as I learn and grow.*
```

**Backend Step 4: Add journaling protocol to AGENTS.md**
Insert after "## Memory" section:
```markdown
### 🔄 Real-Time Journaling Protocol (CRITICAL!)

**BEFORE any browser work or token-heavy tasks:**

1. **Write the plan to today's journal** (`memory/YYYY-MM-DD.md`)
2. **List ALL the steps** you're about to execute
3. **Mark status:** STARTING / IN PROGRESS / COMPLETE / FAILED

**Example:**
\`\`\`markdown
## 11:30 AM - [Task Name]

**Goal:** [What you're trying to do]

**Steps:**
1. Step one
2. Step two
3. Step three

**Status:** STARTING...
\`\`\`

**DURING the work:**
- Update status as you complete each step
- If you extract info from browser: WRITE IT TO FILE immediately
- ONE snapshot → extract → save → close browser

**AFTER (success or fail):**
\`\`\`markdown
**Status:** ✓ COMPLETE - [Result/URL]
OR
**Status:** ✗ FAILED - Reason: [What went wrong]
\`\`\`

**Why this matters:**
- Browser snapshots eat 50-150k tokens EACH
- Context compaction = you forget what you were doing
- When user says "try again" you have the STEPS to follow
- No asking "what was I doing?" or "how do I do this?"

**The rule:** If it uses browser or risks token overflow → JOURNAL THE STEPS FIRST
```

### Key Rules:
1. WAIT for user input at each step before proceeding
2. Backend action happens AFTER receiving input
3. BOOTSTRAP.md must be self-contained for fresh system
4. Journal EVERYTHING as we work - don't wait to be asked
